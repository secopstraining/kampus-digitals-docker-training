📖 Teori (Hızlı Geçiş)
Docker Nedir? Container teknolojisi kullanan bir platform. Uygulamaları ve bağımlılıklarını izole edilmiş ortamlarda çalıştırır.
VM vs Container:
Virtual Machine:
┌─────────────────────┐
│   App A   │  App B  │
├───────────┴─────────┤
│   OS A    │  OS B   │
├─────────────────────┤
│    Hypervisor       │
├─────────────────────┤
│    Host OS          │
└─────────────────────┘

Container:
┌─────────────────────┐
│   App A   │  App B  │
├───────────┴─────────┤
│  Docker Engine      │
├─────────────────────┤
│    Host OS          │
└─────────────────────┘
Avantajlar:
    • Hafif (MB'lar vs GB'lar)
    • Hızlı başlatma (saniyeler vs dakikalar)
    • Tutarlı çalışma ortamı ("works on my machine" sorunu yok)
    • Resource verimli
Docker Mimarisi:
Docker CLI (docker command)
    ↓
Docker Daemon (dockerd)
    ↓
containerd
    ↓
runc (container runtime)

1.1 Docker Kurulumu (Ubuntu)
ADIM 1: Eski Versiyonları Temizle
# Eski Docker versiyonlarını kaldır
sudo apt remove docker docker-engine docker.io containerd runc 2>/dev/null

# Sistem güncellemesi
sudo apt update
sudo apt upgrade -y

ADIM 2: Gerekli Paketleri Kur
# Temel araçlar
sudo apt install -y \
    ca-certificates \
    curl \
    gnupg \
    lsb-release

ADIM 3: Docker GPG Key Ekle
# Dizin oluştur
sudo mkdir -p /etc/apt/keyrings

# Docker resmi GPG key (--yes flag ile üzerine yazma izni)
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
    sudo gpg --dearmor --yes -o /etc/apt/keyrings/docker.gpg

# Okuma izni ver
sudo chmod a+r /etc/apt/keyrings/docker.gpg

ADIM 4: Docker Repository Ekle
# Repository ayarla
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

ADIM 5: Docker Engine Kur
# Paket listesini güncelle
sudo apt update

# Docker Engine, CLI ve containerd kur
sudo apt install -y \
    docker-ce \
    docker-ce-cli \
    containerd.io \
    docker-buildx-plugin \
    docker-compose-plugin

# Docker versiyonunu kontrol et
docker --version
# Beklenen: Docker version 24.0.x veya üstü

ADIM 6: Docker Servisini Başlat ve Aktif Et
# Docker servisini başlat
sudo systemctl start docker

# Boot'ta otomatik başlasın
sudo systemctl enable docker

# Servis durumunu kontrol et
sudo systemctl status docker

# Beklenen çıktı:
# ● docker.service - Docker Application Container Engine
#      Active: active (running)

ADIM 7: Kullanıcıyı Docker Grubuna Ekle
# Mevcut kullanıcıyı docker grubuna ekle
sudo usermod -aG docker $USER

# Grup değişikliğini aktif et (veya logout/login yap)
newgrp docker

# Test: sudo olmadan docker komutu
docker ps
# Hata vermiyorsa başarılı!
💡 Ne Yaptık? docker grubu Docker daemon'a socket üzerinden erişim izni verir. Artık sudo kullanmadan Docker komutları çalıştırabiliriz.

💡 Anlama Testi
    1. Docker daemon ve Docker CLI arasındaki fark nedir?
       Cevap: CLI kullanıcı komutlarını alır, daemon arka planda çalışır ve container'ları yönetir. API ile haberleşirler.
    2. docker run hello-world komutunu tekrar çalıştırırsanız ne olur?
       Cevap: Image zaten local'de olduğu için pull yapmaz, direkt çalıştırır. Çok daha hızlıdır.
    3. Container durduğunda ne olur? Veri kaybolur mu?
       Cevap: Container durdurulur ama silinmez. docker ps -a ile görülür. Veri container içinde kalır ama volume kullanmazsan container silindiğinde kaybolur.
    4. Neden sudo usermod -aG docker $USER yapıyoruz?
       Cevap: Docker daemon'ı root olarak çalışır. Docker grubuna ekleyerek sudo olmadan erişim sağlıyoruz.

📖 Teori (Hızlı Geçiş)
Container Lifecycle:
docker run    → Create + Start
docker create → Sadece oluştur
docker start  → Durdurulmuş container'ı başlat
docker stop   → Graceful shutdown (SIGTERM)
docker kill   → Zorla durdur (SIGKILL)
docker restart→ Stop + Start
docker rm     → Container'ı sil
Image Lifecycle:
docker pull   → Registry'den indir
docker build  → Dockerfile'dan oluştur
docker push   → Registry'e yükle
docker rmi    → Image'ı sil
docker tag    → Image'a tag ekle
Container States:
Created → Running → Paused → Stopped → Removed
          ↓
       Exited

2.1 Container Çalıştırma (docker run)
Basit Çalıştırma:
# Nginx web sunucusu çalıştır
docker run nginx

# Ne oldu?
# - nginx image pull edildi
# - Container oluşturuldu ve başlatıldı
# - Ön planda çalışıyor (terminal kilitli)
# - Ctrl+C ile durdur
❌ Sorun: Terminal kilitli, başka komut çalıştıramıyoruz!

Detached Mode (Arka Plan):
# -d flag: detached (arka planda)
docker run -d nginx

# Çıktı: Container ID (12 karakter)
# Örnek: a1b2c3d4e5f6

# Container çalışıyor mu?
docker ps

# Çıktı:
# CONTAINER ID   IMAGE    COMMAND                  STATUS        PORTS
# a1b2c3d4e5f6   nginx    "/docker-entrypoint.…"   Up 10 sec     80/tcp

Port Mapping:
# Problem: Nginx 80 portunda ama host'tan erişemiyoruz

# -p flag: port mapping (host:container)
docker run -d -p 8080:80 nginx

# Test et
curl http://localhost:8080

# Çıktı: Nginx welcome page HTML!
💡 Port Mapping Mantığı:
localhost:8080  →  Docker Engine  →  Container:80
   (Host)              (Bridge)         (Nginx)

İsimlendirme:
# --name flag: container'a isim ver
docker run -d -p 8081:80 --name my-nginx nginx

# Artık ID yerine isim kullanabilirsin
docker stop my-nginx
docker start my-nginx
docker rm my-nginx

2.2 Container Yönetimi
Container Listeleme:
# Çalışan container'lar
docker ps

# Tüm container'lar (durmuş olanlar dahil)
docker ps -a

# Sadece container ID'leri
docker ps -q

# Son oluşturulan container
docker ps -l

Container Durdurma:
# Graceful stop (10 saniye timeout)
docker stop my-nginx

# Zorla durdur
docker kill my-nginx

# Birden fazla container
docker stop container1 container2 container3

# Tüm çalışan container'ları durdur
docker stop $(docker ps -q)

Container Silme:
# Durmuş container sil
docker rm my-nginx

# Çalışan container'ı zorla sil
docker rm -f my-nginx

# Tüm durmuş container'ları sil
docker container prune

# Dikkat: Bu işlem geri alınamaz!

2.3 Container İnceleme ve Debug
Log Görüntüleme:
# Container loglarını göster
docker logs my-nginx

# Canlı log izleme (-f: follow)
docker logs -f my-nginx

# Son 10 satır
docker logs --tail 10 my-nginx

# Timestamp ekle
docker logs -t my-nginx

Container İçine Giriş:
# Exec: Çalışan container'da komut çalıştır
docker exec -it my-nginx bash

# Container içindesiniz!
# nginx konfigürasyonunu görün:
cat /etc/nginx/nginx.conf

# Process listesi
ps aux

# Çıkış
exit
💡 -it Flags:
    • -i: Interactive (stdin açık)
    • -t: TTY (terminal emulation)

Container Detayları:
# Tüm metadata JSON formatında
docker inspect my-nginx

# Spesifik bilgi çek (IP adresi)
docker inspect -f '{{.NetworkSettings.IPAddress}}' my-nginx

# Port mapping
docker port my-nginx

# Resource kullanımı (real-time)
docker stats my-nginx

# Çıktı: CPU %, Memory, Network I/O

💡 Anlama Testi
    1. docker run vs docker start farkı nedir?
       Cevap: docker run: Yeni container oluşturur ve başlatır. docker start: Var olan durmuş container'ı başlatır.
    2. Container durduğunda veri kaybolur mu?
       Cevap: Hayır. Container durmuş ama silinmemiş ise veri container içinde kalır. docker rm ile silindiğinde kaybolur (volume kullanılmamışsa).

       Cevap: Host'un 8080 portunu container'ın 80 portuna yönlendir. Dışarıdan 8080'e gelen istekler container'ın 80'ine gider.
    4. Neden docker exec kullanırız?
       Cevap: Çalışan container içinde komut çalıştırmak için. Debug, konfigürasyon kontrolü, log inceleme gibi.

📖 Teori (Hızlı Geçiş)
Dockerfile Nedir? Image oluşturmak için kullanılan text dosyası. Her satır bir "instruction" (komut) içerir.
Image Layers (Katmanlar):
┌─────────────────────┐
│   Python App        │  ← Layer 5 (CMD)
├─────────────────────┤
│   requirements.txt  │  ← Layer 4 (COPY)
├─────────────────────┤
│   pip install       │  ← Layer 3 (RUN)
├─────────────────────┤
│   Python 3.11       │  ← Layer 2 (FROM)
├─────────────────────┤
│   Base OS           │  ← Layer 1
└─────────────────────┘
Her layer cache'lenir. Değişmeyen layer'lar yeniden build edilmez (hız!).
Temel Dockerfile Komutları:
Komut       Açıklama                          Örnek
FROM        Base image                        FROM ubuntu:22.04
RUN         Build sırasında komut çalıştır    RUN apt update
COPY        Dosya kopyala                     COPY app.py /app/
ADD         COPY + extract (tar, url)         ADD file.tar.gz /app/
WORKDIR     Çalışma dizini                    WORKDIR /app
ENV         Environment variable              ENV PORT=8080
EXPOSE      Port bilgisi (dokümantasyon)      EXPOSE 80
CMD         Container başlangıç komutu        CMD ["python", "app.py"]
ENTRYPOINT  Ana process                       ENTRYPOINT ["nginx"]

3.1 İlk Dockerfile: Basit Web Uygulaması
Senaryo: Python Flask web uygulaması container'ına alacağız.
ADIM 1: Proje Dizini Oluştur
# Proje dizini
mkdir -p ~/TechFlow_Docker_Project/03-Images/flask-app
cd ~/TechFlow_Docker_Project/03-Images/flask-app

ADIM 2: Python Flask Uygulaması
# app.py dosyası oluştur
cat > app.py << 'EOF'
from flask import Flask
import os

app = Flask(__name__)

@app.route('/')
def hello():
    return f"""
    <h1>TechFlow Flask App</h1>
    <p>Container ID: {os.uname().nodename}</p>
    <p>Python Version: {os.sys.version}</p>
    <p>Environment: {os.getenv('ENV', 'production')}</p>
    """

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
EOF

ADIM 3: Requirements Dosyası
# requirements.txt oluştur
cat > requirements.txt << 'EOF'
Flask==3.0.0
Werkzeug==3.0.1
EOF

ADIM 4: Dockerfile Yaz
# Dockerfile oluştur
cat > Dockerfile << 'EOF'
# Base image: Python 3.11 slim (küçük boyut)
FROM python:3.11-slim

# Metadata (iyi pratik)
LABEL maintainer="techflow@example.com"
LABEL version="1.0"
LABEL description="TechFlow Flask Web Application"

# Çalışma dizini oluştur
WORKDIR /app

# Requirements'ı kopyala (layer caching için önce bu)
COPY requirements.txt .

# Python bağımlılıklarını kur
RUN pip install --no-cache-dir -r requirements.txt

# Uygulama dosyalarını kopyala
COPY app.py .

# Port bilgisi (dokümantasyon)
EXPOSE 5000

# Environment variable
ENV ENV=production

# Container başlatma komutu
CMD ["python", "app.py"]
EOF
💡 Dockerfile Açıklaması:
    1. FROM: Python 3.11-slim base image (debian tabanlı, küçük boyut)
    2. LABEL: Metadata ekle (opsiyonel ama önerilen)
    3. WORKDIR: /app dizinini oluştur ve oraya geç
    4. COPY requirements.txt: Sadece requirements'ı kopyala (cache için)
    5. RUN pip install: Bağımlılıkları kur
    6. COPY app.py: Uygulama kodunu kopyala
    7. EXPOSE 5000: Port 5000 kullanılacak (bilgilendirme)
    8. ENV: Ortam değişkeni tanımla
    9. CMD: Container başlatınca python app.py çalıştır

3.2 Dockerfile Best Practices
1. Layer Caching Optimizasyonu
❌ Kötü Pratik:
COPY . .  # Her değişiklikte tüm dosyalar kopyalanır
RUN pip install -r requirements.txt  # Her seferinde yeniden install
✅ İyi Pratik:
COPY requirements.txt .
RUN pip install -r requirements.txt  # Cache'lenir
COPY . .  # Sadece kod değişirse bu layer yenilenir

2. RUN Komutlarını Birleştir
❌ Kötü (3 layer):
RUN apt update
RUN apt install -y curl
RUN apt clean
✅ İyi (1 layer):
RUN apt update && \
    apt install -y curl && \
    apt clean && \
    rm -rf /var/lib/apt/lists/*

3. Multi-Stage Build (İleri Seviye)
# Stage 1: Builder (geliştirme araçları ile)
FROM python:3.11 AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --user -r requirements.txt

# Stage 2: Runtime (sadece gerekli dosyalar)
FROM python:3.11-slim
WORKDIR /app
COPY --from=builder /root/.local /root/.local
COPY app.py .
ENV PATH=/root/.local/bin:$PATH
CMD ["python", "app.py"]
Avantaj:
    • Builder stage: Büyük image (compiler, dev tools)
    • Runtime stage: Küçük image (sadece runtime)
    • Final image çok küçük (50-70% azalma)

💡 Anlama Testi
    1. FROM python:3.11 vs python:3.11-slim farkı nedir?
       Cevap: slim: Minimum bağımlılık, küçük boyut (~150MB). full: Daha fazla araç, büyük boyut (~900MB). Production'da slim tercih edilir.
    2. Neden requirements.txt'yi app.py'den önce COPY ediyoruz?
       Cevap: Layer caching! app.py değiştiğinde requirements layer'ı cache'ten gelir, pip install tekrar çalışmaz.
    3. CMD vs ENTRYPOINT farkı nedir?
       Cevap: CMD: Override edilebilir. ENTRYPOINT: Container'ın ana process'i, override edilemez (veya --entrypoint ile).
    4. Multi-stage build'in avantajı nedir?
       Cevap: Builder stage'de compiler/dev tools kullan, runtime stage'e sadece binary/runtime kopyala. Sonuç: Çok küçük image.

services:
  # MySQL Database
  db:

    restart: always

      MYSQL_ROOT_PASSWORD: rootpass123
      MYSQL_DATABASE: wordpress

    image: wordpress:latest

      WORDPRESS_DB_PASSWORD: wppass123

volumes:

  wp_data:

networks:
  wp-network:

EOF

        ◦ ports: Host 8080 → Container 80

        ◦ WORDPRESS_DB_HOST: db: Service ismi ile DNS çözümlenir!
volumes: Named volume tanımları (Docker yönetir)
networks: Custom network (izolasyon)

📖 Teori (Hızlı Geçiş)
Container Storage Katmanları:
┌─────────────────────────┐
│ Container Layer (R/W)   │ ← Geçici, silinebilir
├─────────────────────────┤
│ Image Layers (R/O)      │ ← Kalıcı, paylaşımlı
└─────────────────────────┘
Sorun: Container silindiğinde veriler kaybolur!
Çözüm: Volumes
Volume Türleri:
    1. Named Volume (Docker yönetir)
docker volume create my-vol
docker run -v my-vol:/data nginx
    2. Anonymous Volume (Otomatik oluşur)
docker run -v /data nginx
    3. Bind Mount (Host path)
docker run -v /host/path:/container/path nginx
Karşılaştırma:
Özellik         Named Volume    Anonymous    Bind Mount
Yönetim         Docker          Docker       Manuel
Backup          Kolay           Zor          Kolay
Performans      Yüksek          Yüksek       Orta
Path            Docker seçer    Docker seçer Sen belirlersin
Production      ✅ Önerilir     ❌ Önerilmez ⚠️ Dikkatli

⚠️ Sık Yapılan Hatalar
Hata                    Sebep                    Çözüm
Volume boş              Yanlış path              Container içi path kontrol et
Permission denied       User/group uyuşmuyor     chmod veya Dockerfile'da USER

💡 Anlama Testi
    1. Anonymous volume ne zaman kullanılır?
       Cevap: Dockerfile'da VOLUME /data varsa otomatik oluşur. Genelde kullanma, named volume tercih et.
    2. Bind mount production'da neden önerilmez?
       Cevap: Host filesystem'e bağımlı, portable değil, permission sorunları, güvenlik riski.
    3. Volume driver nedir?
       Cevap: Volume'u nerede saklayacağını belirler. local (default), nfs, cloud (AWS EBS, Azure Disk) olabilir.

from flask import Flask, jsonify
import os
import redis

app = Flask(__name__)

# Redis connection
redis_client = redis.Redis(
    host=os.getenv('REDIS_HOST', 'redis'),
    port=6379,
    decode_responses=True
)

# Sample products
PRODUCTS = [
    {"id": 1, "name": "Laptop", "price": 999.99, "stock": 50},
    {"id": 2, "name": "Smartphone", "price": 699.99, "stock": 100},
    {"id": 3, "name": "Headphones", "price": 149.99, "stock": 200},
    {"id": 4, "name": "Keyboard", "price": 79.99, "stock": 150},
    {"id": 5, "name": "Mouse", "price": 49.99, "stock": 300},
]

@app.route('/health')
def health():
    return jsonify({"status": "healthy", "service": "product-service"})

@app.route('/api/products')
def get_products():
    # Try to get from cache
    cached = redis_client.get('products')
    if cached:
        return jsonify({"source": "cache", "products": eval(cached)})

    # Cache miss - store in Redis
    redis_client.setex('products', 60, str(PRODUCTS))
    return jsonify({"source": "database", "products": PRODUCTS})

@app.route('/api/products/<int:product_id>')
def get_product(product_id):
    product = next((p for p in PRODUCTS if p['id'] == product_id), None)
    if product:
        return jsonify(product)
    return jsonify({"error": "Product not found"}), 404

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=5000)
EOF

const express = require('express');
const { Pool } = require('pg');

const app = express();
app.use(express.json());

const pool = new Pool({
    host: process.env.DB_HOST || 'db',
    port: 5432,
    user: process.env.DB_USER || 'techflow',
    password: process.env.DB_PASS || 'techflow123',
    database: process.env.DB_NAME || 'orders'
});

// Health check
app.get('/health', (req, res) => {
    res.json({ status: 'healthy', service: 'order-service' });
});

// Get all orders
app.get('/api/orders', async (req, res) => {
    try {
        const result = await pool.query('SELECT * FROM orders ORDER BY created_at DESC LIMIT 50');
        res.json(result.rows);
    } catch (err) {
        res.status(500).json({ error: err.message });
    }
});

// Create order
app.post('/api/orders', async (req, res) => {
    const { customer_name, product_id, quantity } = req.body;
    try {
        const result = await pool.query(
            'INSERT INTO orders (customer_name, product_id, quantity) VALUES ($1, $2, $3) RETURNING *',
            [customer_name, product_id, quantity]
        );
        res.status(201).json(result.rows[0]);
    } catch (err) {
        res.status(500).json({ error: err.message });
    }
});

const port = process.env.PORT || 5001;
app.listen(port, () => {
    console.log(`Order service running on port ${port}`);
});
